# Mimo TTS

## Endpoints

- `POST /api/ip/v1/mimo/tts/tasks`: synthesize immediately and return a task-shaped completed result for compatibility.
- `GET /api/ip/v1/mimo/tts/tasks/{taskId}`: return completed compatibility status for a known task id; it does not fetch persisted audio.
- `POST /api/ip/v1/mimo/tts/synthesize`: synthesize immediately and require audio content in the response.

Legacy aliases under `/api/ip/v1/xunfei/tts/**` are still mapped by the backend for older frontend code, but new code should use `/mimo/tts/**`.

## Lifecycle

1. `createTask` calls Mimo TTS synchronously and returns `taskStatus=5` plus `audioBase64`.
2. `queryTask` is a compatibility endpoint; it does not store or recover audio generated by the create call.
3. `synthesizeAndWait` calls the same Mimo TTS path and fails if no audio content is returned.
4. Audio is usable when `audioBase64` or `audioUrl` is present; new Mimo responses normally use `audioBase64`.

## Key Fields

- `taskId`: compatibility identifier from Mimo response id or local fallback.
- `taskStatus`: compatibility status; Mimo success is mapped to completed status `5`.
- `audioBase64` / `audioUrl`: final audio result.
- `pybufContent` / `pybufUrl`: legacy response aliases normalized by frontend service.
